# As-Is Architecture — Java Pet Store 1.3.1

A snapshot of the legacy system as inventoried from the 1.3.1 source distribution, to anchor the
migration. The goal of this document is **understanding before changing**: know the seams,
dependencies, and data model before touching anything.

## 1. Deployment topology (J2EE 1.3)

The distribution ships **four** enterprise archives, deployed to a J2EE 1.3 app server
(reference target: J2EE RI / later JBoss):

| EAR                 | Role                                                        | In scope? |
|---------------------|-------------------------------------------------------------|-----------|
| `petstore.ear`      | Consumer **storefront** (browse, cart, account, checkout)   | ✅ **yes** |
| `petstoreadmin.ear` | Admin console (order management, Swing/JNLP client)          | ❌ no |
| `supplier.ear`      | Supplier fulfilment + inventory                             | ❌ no |
| `opc.ear`           | Order Processing Center — receives orders asynchronously    | ❌ no (seam) |

The storefront communicates with the OPC **asynchronously via JMS**: at checkout it serializes
a purchase order to XML and drops it on a queue. The OPC, supplier, and admin apps form a
choreographed back-office. **We migrate only the storefront** and replace the JMS hand-off with
a documented outbound port (see [ADR-0006](../adr/0006-order-submission-port-replaces-jms-opc.md)).

## 2. Storefront internal architecture

Roughly 309 Java source files across the distribution; the storefront app itself is ~51 classes
plus shared components and a ~54-class in-house web framework.

```
Browser
  │  HTTP
  ▼
WAF "Web Application Framework"  (src/waf) ── custom MVC, the Struts-era ancestor
  • MainServlet (front controller)          • ScreenFlowManager (screen-flow.xml)
  • RequestProcessor → HTMLAction            • TemplateServlet / JSP views
  │
  ▼  fires request events
Web-tier controller  (petstore/controller/web)
  • ShoppingWebController → delegates to EJB tier
  │
  ▼  EJB local calls
EJB-tier controller  (petstore/controller/ejb)
  • ShoppingClientControllerEJB (stateful session bean)
  • StateMachine + EJBActions (event → action)
  │
  ▼  uses reusable business components (EJBs)
Business components  (src/components/*)
  • catalog        (DAO-based, read-only browse)
  • cart           (stateful session cart)
  • customer/account/contactinfo/address/creditcard  (CMP entity beans)
  • signon         (auth: username/password, CMP User entity + servlet filter)
  • purchaseorder / lineitem  (order model, serialized to XML → OPC)
  • asyncsender    (JMS producer to OPC)
  • uidgen, mailer, servicelocator, xmldocuments, processmanager
  │
  ▼  JDBC / CMP
Relational DB (Cloudscape/Derby by default; Oracle DDL also shipped)
```

### Key patterns in play (and their modern replacements)

| Legacy pattern | Where | Modernized to |
|----------------|-------|---------------|
| Front controller + screen flow | WAF `MainServlet`, `screen-flow.xml` | Spring MVC `@Controller` + view resolution |
| Service Locator / JNDI lookups | `servicelocator` component | Spring dependency injection |
| Business Delegate / Session Facade | `ShoppingClientController` EJB | `@Service` application services |
| DAO + DAO Factory (per-DB) | `catalog.dao.*` (`CloudscapeCatalogDAO`) | Spring Data repositories behind ports |
| CMP entity beans | customer/account/signon | JPA entities (+ Mongo documents) |
| Container-managed auth + `SignOnFilter` | `signon` component | Spring Security |
| Async JMS to OPC | `asyncsender` | `OrderSubmissionPort` (outbound port + adapter) |
| Value Object / Transfer Object | `*Model`, XML docs | Domain objects / DTOs |

## 3. Data model (relational)

The catalog schema is defined declaratively in
`src/apps/petstore/src/docroot/populate/PopulateSQL.xml` (DDL for Cloudscape **and** Oracle),
and seeded from `Populate-UTF8.xml`. The defining characteristic is **i18n via `_details`
side-tables** keyed by `locale`:

```
category(catid PK)
category_details(catid, locale PK → name, image, descn)
product(productid PK, catid FK)
product_details(productid, locale PK → name, image, descn)
item(itemid PK, productid FK)
item_details(itemid, locale PK → listprice, unitcost, image, descn, attr1..attr5)
```

Customer/account/sign-on data uses **CMP entity beans** (schema generated by the container
rather than shipped DDL): `User(username, password)`, plus `Account`, `ContactInfo`, `Address`,
`CreditCard`, `ProfileInfo`. Orders are **not persisted by the storefront** — they are streamed
to the OPC. (Our migration *adds* order persistence; see the target architecture.)

## 4. Build & runtime

- **Build:** Ant (`build.xml` per module) producing EARs/WARs/JARs; JSTL + custom tag libs.
- **Views:** JSP + tag libraries under `docroot/`.
- **Runtime:** J2EE 1.3 container, `javax.*` APIs, container-managed transactions & security.

## 5. Migration seams identified

1. **DB access is already abstracted** behind DAOs/CMP — a natural port boundary.
2. **OPC integration is already asynchronous & message-based** — a clean anti-corruption seam;
   we stub it without touching storefront logic.
3. **Catalog is read-only** — the lowest-risk slice to migrate first and prove the pipeline.
4. **Sign-on is a self-contained component** — swap for Spring Security in isolation.
5. **i18n `_details` tables** are the trickiest data-model detail to preserve faithfully.
